# ⚡ AI Логер та аналізатор промптів

Професійний веб-застосунок для зберігання, версіонування, оцінювання та AI-аналізу промптів. Дозволяє розробникам створювати власну або командну базу знань промптів, відстежувати їхні зміни (версії) та отримувати миттєві рекомендації щодо покращення через Gemini API.

---

## 🚀 Основні можливості

- **Додавання промптів** з категорією (`debugging`, `refactoring`, `docs`, `testing`, `other`) та цільовою моделлю (`gpt-4`, `claude`, `gemini`).
- **Версіонування (v1 → v2 → v3)**: при редагуванні стара версія промпту не перезаписується, а створюється нова.
- **Оцінювання результатів**: можливість виставити кожній версії оцінку (1–5 ⭐) за допомогою інтерактивної форми.
- **AI-аналіз ефективності**: інтегрований модуль аудиту, що використовує Gemini 3.5 Flash API для детального розбору промпту, виявлення сильних та слабких сторін і надання оптимізованого варіанту.
- **Гнучка фільтрація**: пошук та фільтрація списку промптів за категорією та моделлю.
- **Інтерактивний лог аудиту**: відстеження всіх дій (створення, додавання версії, оцінювання) у реальному часі на базі тригерів у БД.

---

## 🏗️ Архітектура бази даних (PostgreSQL)

Проєкт використовує реляційну базу даних із наступною структурою:

### Таблиця `prompts`
Зберігає основну інформацію про промпт.
- `id` (SERIAL PRIMARY KEY)
- `title` (VARCHAR) — заголовок промпту.
- `category` (VARCHAR) — категорія (`debugging`, `refactoring`, `docs`, `testing`, `other`).
- `target_model` (VARCHAR) — модель (`gpt-4`, `claude`, `gemini`).
- `created_at` (TIMESTAMP) — дата створення.

### Таблиця `prompt_versions`
Зберігає конкретні текстові версії та оцінки.
- `id` (SERIAL PRIMARY KEY)
- `prompt_id` (INTEGER REFERENCES prompts(id) ON DELETE CASCADE) — зв'язок з промптом.
- `version_number` (INTEGER) — порядковий номер версії (v1, v2 тощо).
- `content` (TEXT) — текст промпту.
- `rating` (INTEGER CHECK (1..5)) — оцінка ефективності користувачем.
- `ai_feedback` (TEXT) — структурований HTML-аналіз від AI.
- `created_at` (TIMESTAMP) — дата створення версії.

### Таблиця `prompt_audit_log`
Журнал подій для аудиту.
- `id` (SERIAL PRIMARY KEY)
- `prompt_id` (INTEGER) — ID промпту.
- `action` (VARCHAR) — дія (`created`, `version_added`, `rated`).
- `happened_at` (TIMESTAMP) — дата та час події.

## 🏆 Реалізація ключових технологій БД (3 принципи)

### 1. Тригери (Triggers) ✅
У базі даних налаштовано два тригери:
- **Автоматичне призначення номерів версій (`trigger_set_version_number`)**: Перед додаванням нового запису до `prompt_versions` виконується функція `set_prompt_version_number()`. Вона шукає максимальний номер версії для даного `prompt_id` і присвоює наступний `MAX + 1` (починаючи з 1). Вся логіка версіонування реалізована на рівні БД і не перевантажує Flask.
- **Автоматичне логування подій (`trigger_log_prompt_action`)**: Після створення нової версії або оновлення рейтингу автоматично створюється запис у таблиці `prompt_audit_log` із відповідною дією (`created`, `version_added`, або `rated` при зміні оцінки).

### 2. Транзакції (Transactions) ✅
При додаванні нового промпту у файлі [prompts.py](file:///C:/Users/user/Documents/ШАРАГА/post/promt_log/routes/prompts.py) відкривається транзакція.
Створюється запис у таблиці `prompts`, і одразу після цього додається перша версія в `prompt_versions`. Якщо будь-який крок викликає помилку (наприклад, перевищення ліміту символів чи некоректне значення), Flask виконує `db.session.rollback()`. Це гарантує відсутність «сирітських» записів у базі даних (коли промпт створений, а версія до нього — ні).

### 3. Збережені процедури (Stored Procedures) ✅
Для оптимізації роботи з базою даних створена функція `get_best_prompt_version(prompt_id)` на мові PL/pgSQL:
```sql
CREATE OR REPLACE FUNCTION get_best_prompt_version(p_id INT)
RETURNS SETOF prompt_versions AS $$
BEGIN
    RETURN QUERY
    SELECT * FROM prompt_versions
    WHERE prompt_id = p_id
    ORDER BY rating DESC NULLS LAST, version_number DESC
    LIMIT 1;
END;
$$ LANGUAGE plpgsql;
```
Ця процедура повертає версію промпту з найвищою оцінкою користувача. Якщо оцінки однакові — повертається найновіша версія з цим рейтингом. Виклик процедури здійснюється безпосередньо з Flask за допомогою raw SQL запиту: `SELECT * FROM get_best_prompt_version(:id)`.

## ⚙️ Налаштування та запуск

### 1. Встановлення залежностей
Переконайтеся, що активовано віртуальне оточення `venv`, та встановіть бібліотеки:
```bash
pip install -r requirements.txt
```

### 2. Налаштування середовища (`.env`)
Створіть або оновіть файл `.env` у корені проєкту:
```env
FLASK_APP=app.py
FLASK_ENV=development
SECRET_KEY=your_secret_key_here
DATABASE_URL=postgresql://postgres:password@localhost:5432/prompt_logger
GEMINI_API_KEY=your_gemini_api_key_here
```

### 3. Ініціалізація бази даних
Для швидкого створення таблиць, тригерів та збереженої процедури запустіть скрипт ініціалізації:
```bash
python init_db.py
```

### 4. Запуск веб-застосунку
Запустіть Flask сервер:
```bash
flask run --port=5000
```
Застосунок буде доступний за адресою [http://localhost:5000](http://localhost:5000).
